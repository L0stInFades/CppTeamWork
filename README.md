## 基准测试方法

所有测试由 `benchmark.sh` 自动执行，关键设计：

1. **隔离性** —— 每次测试在独立临时目录运行，不共享数据文件
2. **预热** —— 首次运行不计入统计，消除文件系统缓存冷启动偏差
3. **重复取中位数** —— 编译和内存测 5 次，启动和批量测 10 次，取中位数而非平均值（抗干扰更强）
4. **公平编译** —— 每次编译前清除全部产物，Rust 执行 `cargo clean`，Zig 删除 `zig-out` 和 `.zig-cache`
5. **毫秒精度** —— 使用 `python3 time.time()` 获取毫秒级时间戳

## 维度一：编译时间

```
Go          ██ 349ms
TypeScript  ██████████████ 1409ms
C++         ███████████████ 1537ms
Rust        ██████████████████████████ 2693ms
Zig         █████████████████████████████████████████████████████████████████████████████████████████████████████████ 27295ms
```

### 分析

**Go 以 349ms 碾压全场**，这得益于 Go 编译器的独特设计哲学——它从诞生之初就把编译速度作为一等公民。Go 编译器是单遍编译（single-pass），不做重量级优化，依赖链管理极简，整个过程几乎没有冗余。

**C++ 和 TypeScript 处于第二梯队**（~1.4-1.5s）。C++ 在这个项目中只有单文件，没有头文件地狱的问题，所以表现尚可。如果引入 STL 模板和多文件编译，时间会急剧膨胀。TypeScript 的 `tsc` 实际上做的是类型检查 + 转译，没有代码生成和链接阶段，所以也比较快。

**Rust 约 2.7s**，考虑到 `cargo clean` 后需要编译依赖 `libc` crate + 链接，这个成绩已经不错。增量编译会大幅降低这个时间，但本测试故意每次 clean 以测量全量编译。

**Zig 最慢（27s）**，原因是 Zig 的构建系统在 `ReleaseFast` 模式下会重新编译整个标准库。Zig 目前没有预编译标准库缓存的机制，每次 clean build 都要从零开始。这是 Zig 生态成熟度的体现，随着编译器迭代会改善。

### 启示

> 如果你的项目迭代速度极高、需要频繁重编，**Go 的编译速度优势是质的差距**——它快到让编译这件事几乎不存在。

## 维度二：产物体积

```
TypeScript  █ 31KB (JS bundle, 需要 Node.js)
C++         ██ 60KB → strip 后 56KB
Zig         █████████████ 520KB → strip 后 466KB
Rust        ████████████ 490KB → strip 后 413KB
Go          ██████████████████████████████████████████████████████ 2671KB → strip 后 2552KB
```

### 分析

**C++ 以 60KB 夺冠**（原生可执行文件中最小）。C++ 通过动态链接 libc 和 libstdc++，把运行时库的体积外包给了系统。这也意味着它对系统环境有隐式依赖——换一台没有对应版本 libstdc++ 的机器可能跑不起来。

**Zig（520KB）和 Rust（490KB）体量接近**，两者都默认静态链接标准库。strip 后分别降到 466KB 和 413KB，去掉的主要是调试符号。这两个大小对嵌入式或容器场景来说仍然非常友好。

**Go 最大（2.6MB）**，因为 Go 是完全静态链接，把整个 Go runtime（goroutine 调度器、GC、网络 poller 等）都打包进去了。好处是"一个文件走天下"，交叉编译后直接扔到目标机器就能跑，不需要任何依赖。

**TypeScript 的 31KB** 是 JS 源码体积，不能直接与编译型语言对比。运行它需要整个 Node.js 引擎（~80MB），加上 `node_modules` 依赖。实际部署体积远大于这个数字。

### 启示

> 产物大小是**链接策略**的直接映射：动态链接（C++）最小但依赖多，完全静态链接（Go）最大但零依赖。选择取决于部署场景。

## 维度三：冷启动时间

```
Zig         ████████████████████████████ 58ms
C++         █████████████████████████████ 59ms
Rust        █████████████████████████████ 59ms
Go          █████████████████████████████ 59ms
TypeScript  ██████████████████████████████████████████████████████████████████████████████████████████████████████ 206ms
```

### 分析

**四种编译型语言的冷启动时间完全一致（58-59ms）**，差异在统计误差范围内。这个结果在意料之中——对于一个简单的终端程序，启动过程就是：操作系统加载二进制 → 初始化 BSS/data 段 → 跳到 `main`。无论语言如何，这几步的耗时主要由操作系统决定。

注意：这里的 ~58ms 中大部分是 shell fork + exec 的开销和 python3 计时的调用开销，真实进程启动时间应在微秒级别。但由于所有语言共用同一套计时方式，横向对比仍然公平。

**TypeScript 慢 3.5 倍（206ms）**，因为每次启动都需要：
1. 启动 Node.js V8 引擎（~100ms）
2. 解析和编译 JS 代码为字节码
3. JIT 编译热路径

这是解释型 / JIT 语言的固有代价。在 CLI 工具场景下，每次调用都要付这笔启动税。

### 启示

> 如果你在写一个**频繁被调用的 CLI 工具**（如 git hook、编辑器插件），编译型语言的启动优势会被成百上千次放大。TS/Node 更适合长驻进程（如 Web 服务器）。

## 维度四：批量业务性能

测试场景：添加 500 条开销记录 → 查看全部 → 月度统计 → 按年列出 → 保存退出。

```
Zig         █████████████████████████████████ 66ms
C++         ██████████████████████████████████ 69ms
Rust        ███████████████████████████████████ 70ms
Go          ███████████████████████████████████ 70ms
TypeScript  ████████████████████████████████████████████████████████████████████████████████████████████████████████████████████████ 246ms
```

### 分析

**四种编译型语言再次打成平手（66-70ms）**。这个结果揭示了一个重要事实：**对于 I/O 密集 + 轻计算的业务逻辑，语言本身的运行时性能差异可以忽略不计**。瓶颈在于 stdio 交互、字符串解析和文件写入，这些操作最终都落到操作系统的系统调用上，语言层面的差异被系统调用的开销淹没了。

Zig 以 66ms 微弱领先，可能得益于其零开销抽象和更少的运行时初始化。但 4ms 的差距（~6%）在实际使用中无法感知。

**TypeScript 约 246ms，是编译型语言的 3.5 倍**。额外的约 180ms 中：
- ~140ms 是 Node.js 启动开销（与冷启动测试的 206ms vs 59ms 一致）
- ~40ms 是 V8 解释执行的额外开销

换句话说，如果去掉启动时间，TypeScript 的纯业务执行速度大约是编译型语言的 **3-4 倍慢**，但绝对值只多了约 40ms——用户几乎不会感知到。

### 启示

> 在这类轻量业务场景中，**性能不应该成为语言选型的决定因素**。C++/Zig/Rust/Go 的差异在噪声范围内，TS 虽慢但绝对延迟仍在可接受范围。

## 维度五：峰值内存占用

测试场景与维度四相同（500 条记录的批量操作），使用 `/usr/bin/time -l` 采集 `maximum resident set size`。

```
C++         ██ 908KB
Rust        ██ 1052KB
Zig         ███ 1096KB
Go          ██████ 3040KB
TypeScript  █████████████████████████████████████████████████████████████████████████████████████████████████████ 43540KB
```

### 分析

**C++ 最省（908KB）**，不到 1MB。C++ 的内存模型是"你只为你用到的买单"——没有 GC、没有运行时元数据、没有协程栈。`malloc/free` 直接操作堆，开销最低。

**Rust（1052KB）和 Zig（1096KB）接近 C++ 水平**。Rust 的所有权系统在编译期完成内存管理决策，运行时零开销。Zig 同样是无 GC 手动管理。两者多出的约 100-200KB 主要来自它们各自标准库的静态初始化数据。

**Go 约 3MB**，是 C++ 的 3.3 倍。这是 Go runtime 的固定开销：
- GC 元数据和标记位图（每 8 字节堆内存对应 1 bit 标记位）
- Goroutine 栈（主 goroutine 初始 8KB，可动态增长）
- M:N 调度器的内部数据结构
- 内存分配器的 span/cache 层级

对于这个小程序来说 3MB 看起来不少，但在实际服务端场景中，Go 的内存开销会随业务数据线性增长，runtime 的固定开销占比会越来越小。

**TypeScript/Node.js 占用 43MB**，是 C++ 的 **48 倍**。V8 引擎是这个数字的主要贡献者：
- V8 堆（新生代 + 老生代）初始就分配 ~20MB
- JIT 编译产生的机器码缓存 ~10MB
- Node.js 内置模块（fs, path, buffer 等）的内存映射 ~10MB
- JS 对象的 Hidden Class 元数据、属性存储等额外开销

这是 **运行时复杂度** 的代价。V8 的 JIT 编译器虽然让 JS 能接近原生速度运行，但它本身就是一个庞大的程序。

### 启示

> 内存占用是**语言运行时模型**的直接反映。手动管理（C/C++/Zig）< 编译期管理（Rust）< 轻量 GC（Go）<< 重量级 VM（Node.js/V8）。对于内存受限的嵌入式场景，C++/Zig/Rust 是唯一选择。

## 结论与选型建议

### 没有银弹

这次对比实验最重要的结论是：**没有一种语言在所有维度都是最优的**。每种语言的设计哲学决定了它在不同维度上的取舍：

| 语言 | 设计哲学 | 最强维度 | 最弱维度 |
|------|---------|---------|---------|
| C++ | 零开销抽象 | 产物大小、内存 | 编译时间（大项目） |
| Zig | 更好的 C | 运行时性能 | 编译时间 |
| Rust | 安全 × 性能 | 内存安全 + 接近 C 性能 | 编译时间、学习曲线 |
| Go | 简洁实用 | 编译时间 | 产物大小、内存 |
| TypeScript | 开发效率 | 生态、开发速度 | 一切运行时指标 |

### 选型参考

- **嵌入式 / 资源受限**：C++ 或 Zig（内存 < 1.1MB，产物 < 520KB）
- **系统编程 / 安全关键**：Rust（编译期内存安全，性能与 C++ 持平）
- **后端微服务 / DevOps 工具**：Go（349ms 编译、单文件部署、足够好的性能）
- **快速原型 / Web 全栈**：TypeScript（最大生态、最快开发速度，性能完全够用）
- **高频 CLI 工具**：C++ / Rust / Zig / Go（启动时间 ~59ms vs TS 206ms）

### 最终观点

> 对于 99% 的应用场景来说，这五种语言的**性能差异远小于架构设计和算法选择带来的差异**。选择语言时，应优先考虑团队熟悉度、生态成熟度和维护成本，而非微秒级的性能差距。性能对比的价值在于**理解各语言的运行时模型**，而非简单地分出高下。

## 如何复现

```bash
# 克隆仓库
git clone https://github.com/L0stInFades/CppTeamWork.git
cd CppTeamWork

# 确保以下工具链已安装：g++, zig, cargo, go, node/npx
# 运行 benchmark
chmod +x benchmark.sh
./benchmark.sh
```

脚本会自动检测可用的语言工具链，跳过未安装的语言，最终输出 Markdown 格式的对比表格。
